**Amendment (2026-06-09)**: §1's inherent-impl discriminator originally carried a source-order `#<ordinal>` while calling itself "source-order-independent" — a self-contradiction. Option (b) (user-confirmed) **drops the ordinal**: same-`(type, positional-generic-signature, cfg)` inherent impls now merge into one `impl` entity, making the discriminator genuinely reorder- and method-set-stable. cfg-twin impls (now including trait impls) are split by the `@cfg(...)` suffix instead. The `impl` block also becomes its own entity (its methods parent to it). The shared corpus `fixtures/qualnames_rust.json` and the Wardline-vendored copy are amended in lockstep. The Context table below documents the *original draft's* collisions and is left as historical record.
**Amendment (2026-06-09, self-type-args)**: a second collision family was found in the impl-key §2: the self type's name dropped its concrete generic arguments, so `impl Foo<i32>` and `impl Foo<u32>` (and their trait twins `impl Display for Foo<i32>` / `…<u32>`) both rendered the bare-`Foo` key and `seen_impl_ids` spuriously merged them, silently overwriting a like-named method at the writer's `ON CONFLICT(id) DO UPDATE`. The fix folds the self type's **concrete** generic arguments into `<Type>` (`…Foo<i32>` vs `…Foo<u32>`), rendering the impl's **own** declared params positionally (`$0`) so a generic-param rename does not churn and same-instantiation blocks still merge. See §2 below. The shared corpus and the Wardline-vendored copy are amended in lockstep (third dialect amendment; cross-repo `~/wardline` copy must re-sync).
**Amendment (2026-06-10, generic-arg rendering — reserved-char escape + const spacing)**: Two undecided rendering surfaces were raised by Wardline (the second producer) in their amendment-request letter (`~/wardline/docs/integration/2026-06-10-wardline-loomweave-rust-qualname-amendment-requests.md`), **signed off by the decider 2026-06-10**. (1) **Reserved-colon path-typed generic args:** a concrete generic argument that is itself a `::`-path (`impl From<std::io::Error> for Foo` — ubiquitous in real Rust) rendered a colon-bearing locator. Loomweave's `entity_id()` rejected it (`validate_no_colon`) and `degraded_aware` collapsed the *whole cleanly-parsed file* to one `syntax_error` module; Wardline emitted the same bytes un-gated. Neither endpoint was acceptable — the dialect had no canonical form for the construct, and the corpus (single-segment args only) could not see the gap. (2) **Const-generic-arg spacing:** `GenericArgument::Const` was the dialect's one whitespace-bearing rendering (proc-macro2 token spacing: `Foo<- 1>`, `Foo<{ N + 1 }>`), unreproducible from a tree-sitter frontend and inconsistent with `strip_ws` as "the crate's one path/type normaliser". **Decision:** every concrete generic argument — type or const, in both the trait fragment and the self-type prefix — renders through one shared pipeline **`escape_reserved(strip_ws(arg))`** (strip first, then the existing injective `%`→`%25`-then-`:`→`%3A` escape already corpus-pinned on the cfg path): `Foo.impl[From<std%3A%3Aio%3A%3AError>]`, `Foo<std%3A%3Aio%3A%3AError>.impl#<>`, `Foo<{1+2}>.impl#<>`, composing for `Foo<{usize%3A%3AMAX}>`. Collision-free (distinct paths stay distinct), injective/reversible (escape-ordering argument at qualname.rs:308-313), no new escape grammar, rename-stability and `implements`-edge resolution unaffected. **Rejected:** last-segment truncation (re-opens the silent `ON CONFLICT` data-loss family: `io::Error` collides with `fmt::Error`); keeping degrade-whole-file as canonical (one impl erases a file's entity surface; no rendering at all for a ubiquitous construct); Wardline-only normalization (breaks the byte-for-byte second-producer arrangement). **Corpus rows to land with implementation** (extractor-generated): `path_typed_generic_arg_inherent`, `path_typed_generic_arg_trait`, `const_generic_arg_spacing`; the Wardline-vendored copy re-syncs and both producers conform in lockstep (Wardline-side `wardline-be5ee9cc34`, const-spacing half of `wardline-e8f7c0508f`). **Completion (2026-06-11):** the escape was first wired only onto concrete generic args of a `Type::Path` self type; a follow-on (Sprint-4 scale QA) found a **non-`Type::Path` self type** (reference / tuple / slice / raw pointer) carrying a `::`-path — e.g. serde's `impl Serializer for &mut fmt::Formatter` — still fell through `self_ty_locator`'s `type_textual` fallback unescaped and dropped its file. The fallback now routes through the same `escape_reserved(strip_ws(..))` pipeline (`&mut fmt::Formatter` → `&mutfmt%3A%3AFormatter`); corpus row `reference_self_type_path_escape`. A `:`-free fallback (`&Foo`, `(A,B)`) is unchanged (the escape is a no-op). This closes the last reserved-`:` file drop (38/38).
**Amendment (2026-06-11, cfg-twin methods)**: The fifth dialect amendment (following the 2026-06-10 "amendment 4" generic-arg-rendering work), closing the last real-corpus identity defect (`clarion-dfeb905f46`). The `@cfg(...)` discriminant (§1, the cfg-collision policy below) was applied to impl *keys* and to module-level free *items*, but **never to `ImplItem::Fn`** (`extract.rs` built a method qualname `{impl_q}.{name}` without consulting `m.attrs`). So two cfg-gated twin methods that land on the **same impl entity** with the same name — either inside **one** impl block (`impl Foo { #[cfg(unix)] fn go #[cfg(windows)] fn go }`) or across **several blocks that merge** under Option (b) (`impl Foo { #[cfg(unix)] fn go } impl Foo { #[cfg(windows)] fn go }`) — both rendered `…Foo.impl#<>.go`, and the writer's `ON CONFLICT(id) DO UPDATE` silently kept the last (a chimera entity; SEI/Wardline taint then key 2–3 functions as one). Every qualname collision the Sprint-3 scale sweep found over ripgrep/serde/tokio/rust-analyzer (ripgrep 8, serde 4, tokio's `AsyncWrite` `poll_*` twins, …) is this one gap. **Decision:** a cfg-gated twin **method** carries its own `@cfg(...)` suffix **after the method name** (`…Foo.impl#<>.go@cfg(unix)` / `…@cfg(windows)`), exactly as a free item does, composing on top of any impl-level `@cfg` already in the prefix. Twin-ness is computed against the **final** impl qualname (post impl-level cfg; since Amendments 6–7, "final" also means post-S/T — the ladder's stage 4) + method name, so impl-level cfg-twins — already split into distinct impl entities — collect **no** redundant method suffix, while a method-twin *inside* a cfg-twin block still does. **Additive:** only already-colliding method ids change. **Rejected:** a per-method *signature* discriminant (cfg twins have byte-identical signatures, so it cannot split them) and a source-order ordinal (the collision is *within* one merged entity, where there is no stable order). **Corpus rows landed with the implementation** (extractor-generated): `method_cfg_twin_inherent`, `method_cfg_twin_trait`, `method_cfg_twin_cross_merged_block`; the Wardline-vendored copy must re-sync and both producers conform in lockstep — but Wardline's Rust frontend has graduated to its `rc5` release branch, so this re-vendor is an escalation-gated **handoff** (`docs/federation/2026-06-11-rust-qualname-amendment-4-5-changeset.md`), not a push. **Visibility sub-item** (the analyze path emits no finding on a duplicate id — only the `dogfood_uniqueness` test and the `qualname_check` example binary consult `duplicate_ids()`): deferred as `clarion-b19fe90c3e` (needs a protocol or writer-contract change beyond this amendment). **Related, not fixed here:** repeated unnamed `const _` items in one module collide the same way (a name-based qualname with a non-unique name) — closed by **Amendment 9** below.
**Amendment 6 (2026-06-11, self-type written-path qualification)**: Closes `clarion-8ff7f233fa`, a Sprint-4 gold-blocker collision family the Sprint-3 sweep had mis-attributed to cfg-twin methods. The impl locator's `<Type>` base rendered only the **last path segment** of a path-qualified self type, so two impls of the same trait for same-*named* types defined in different modules — tokio `chan.rs:572/596`, `impl Semaphore for bounded::Semaphore` + `impl Semaphore for unbounded::Semaphore`, both rendering `…chan.Semaphore.impl[Semaphore]` — spuriously merged in `seen_impl_ids` and their five like-named method ids (`add_permit`, `add_permits`, `is_idle`, `close`, `is_closed`) silently collapsed at the writer's `ON CONFLICT(id) DO UPDATE` (a chimera entity). **Decision — the residual-collision ladder (normative; Amendment 7 rides it).** Per extraction unit (one `walk_items` item list), impl qualnames are decided in stages, each keyed on the previous stage's output: **(1) `@cfg`** — the Amendment-1/5 machinery, unchanged: impl-level cfg-twin-ness is computed on the **bare pre-cfg keys** exactly as today, so already-`@cfg`-split twins keep their current ids byte-for-byte; **(2) stage S** (this amendment) — impls are grouped by their **post-cfg** qualname, each member's *witness* being its **written self-type path** (for a qself-free `Type::Path`: segment idents joined `::`, a leading `::` contributing a leading separator — mirroring Amendment 7, so `impl Tr for ::a::X` and `impl Tr for a::X` carry distinct witnesses — with **no generic arguments**: args are already normalized by the self-type-args amendment and must not affect the witness; for a non-`Type::Path` or qself-bearing self type: the Amendment-4 textual render, already fully path-qualified). A group with ≥ 2 distinct witnesses re-renders every **qself-free `Type::Path`** member's base as `escape_reserved(witness)` (a leading `::` rendering a leading `%3A%3A`) plus the existing final-segment generic-args machinery verbatim — `{m}.a%3A%3AX.impl[T]`, `{m}.b%3A%3AX<$0>.impl#<>`; a qself-bearing or non-`Type::Path` member keeps the single-escaped Amendment-4 fallback rendering it already has (re-applying the escape would double-escape it). A single-segment witness renders byte-identically to today, so within a fired group only multi-segment members move, and un-fired groups change **nothing**; **(3) stage T** — Amendment 7 below, grouped on post-S qualnames; **(4) method-`@cfg`** — Amendment 5, unchanged mechanics, keyed on the final (post-S/T) impl qualname. The ladder ordering was adjudicated against an *independent gates* alternative precisely for the cfg-cross-path case: `#[cfg(unix)] impl T for a::X` / `#[cfg(windows)] impl T for b::X` are split by stage 1 on the bare key, so S sees singleton groups, stays cold, and the ids stay byte-identical to today's `@cfg`-only form (pinned by the `trait_path_cfg_twin_unqualified` corpus row and the `cross_path_cfg_twins_keep_todays_cfg_ids` stability test). **File-local gating is complete for valid Rust:** two impls can only share a bare qualname if they share a module path, and one module path means one item list — a doubly-defined file module is E0761, a doubly-defined inline module E0428; `include!` is invisible to syn (both engines agree the items do not exist); the one true gap is a module mounted at the same path from **two different files via `#[path]`**, which evaded the file-local gate until **Amendment 8** (below) closed it. A second known residual: identical-written-path coherence-illegal twins (two `impl T for a::X` blocks spelled identically) still collide by construction — no witness can split them — and are reported by `duplicate_ids()`. **Known limitation (as-written sensitivity):** the witness is the path *as written*, so re-spelling a self type through a `use` alias versus its full path churns the id — but **only while a twin exists** (a lone impl never qualifies); this is the same as-written discipline the rest of the dialect already follows, and the price of staying resolution-free. **Rejected:** *always-qualify* (qualify every path-typed self type unconditionally) — mass SEI churn, ~2 % of all impls plus their methods move for zero collisions closed; *dotted `a.X` join* (folding the path into dot-segments) — **aliases a genuine module path**: `mod a { impl X {} }` already owns `foo.a.X.impl#<>`, so the join mints a brand-new collision family instead of closing one; *whole-text witness* (comparing the full self-type rendering, args included) — an Amendment-3 positional rename (`impl<T> … a::X<T>` → `<U>`) falsely fires the gate; *leading-`crate::` normalization* (canonicalizing the written path's head before comparing) — requires knowing what the head resolves to, i.e. name resolution, which the single-file second-producer constraint forbids. **Wardline lockstep:** the vendored corpus copy must re-sync, riding the same escalation-gated rc5 handoff as Amendments 4–5, batched with the 6–9 re-vendor wave. **Corpus rows landed with the implementation** (extractor-generated): `self_type_path_twin_trait`, `self_type_path_twin_inherent`, `self_type_path_lone_unqualified_negative`, `self_type_path_mixed_bare_and_qualified`, plus the ladder rows shared with Amendment 7 (`trait_path_cfg_twin_unqualified`, `impl_ladder_self_splits_before_trait`).
**Amendment 7 (2026-06-11, trait-path written-path qualification)**: Closes `clarion-fa8bcf8731`, the sibling Sprint-4 gold blocker. The `impl[…]` fragment keyed on only the trait path's **last segment** (`impl_disc_for` takes `trait_path.segments.last()`), so two same-*named* traits from different paths implemented for the same self type — tokio_util `compat.rs:218/240`, `impl<T> tokio::io::AsyncRead for Compat<T>` + `impl<T> futures_io::AsyncRead for Compat<T>` (and the `AsyncBufRead`/`AsyncWrite` pairs) — both rendered `Compat<$0>.impl[AsyncRead]` and five method ids (`poll_read`, `poll_fill_buf`, `consume`, `poll_write`, `poll_flush`) silently collapsed. The cfg machinery *detects* this collision but cannot split it: the impls carry no `#[cfg]`, so `cfg_suffix` returns `None`. **Decision:** stage **T** of the Amendment-6 residual-collision ladder. Impls are grouped by their **post-S** qualname; each trait-impl member's rendering-witness is the **trait path as written** — a leading `::` contributes a leading `%3A%3A`, segment idents join by `%3A%3A` (`escape_reserved` over the `::`-joined string, reusing the corpus-pinned escape bytes), and the final segment keeps the existing `trait_generic_args` rendering. A group with ≥ 2 distinct renderings switches **every** member's `impl[…]` fragment to the qualified rendering: `Compat<$0>.impl[tokio%3A%3Aio%3A%3AAsyncRead]` vs `Compat<$0>.impl[futures_io%3A%3AAsyncRead]`. A single-segment written path renders byte-identically to the bare fragment, so in a `Tr` vs `other::Tr` pair only the multi-segment member moves; inherent impls have no trait path and never fire T. Running T **after** S yields minimal qualification: the mixed pair (`impl a::Tr for c::X` / `impl b::Tr for d::X`) is already split by S, so T stays cold and the trait fragments keep their bare last segments (corpus row `impl_ladder_self_splits_before_trait`). `impl_disc_for` itself — the corpus-pinned base form — is untouched; the qualified rendering is a separate entry point (`impl_disc_for_qualified`) applied only inside a fired group. `trait_path_for_lookup` / `implements`-edge resolution already use the full path and are unaffected, as is the SEI signature's last-segment `target` (changing it would be mass churn for zero identity value). The Amendment-6 residuals apply identically here: `#[path]`-aliased twins (closed by Amendment 8 below), identical-written-path twins reported by `duplicate_ids()`, and the as-written sensitivity (alias vs full spelling churns only while a twin exists). **Rejected:** *always-qualify* — 6–41 % of trait impls churn per corpus measurement, against a handful of real collisions; *resolving the written path to a canonical crate-rooted form* — needs name resolution, violating the single-file second-producer constraint (Wardline's tree-sitter frontend must reproduce the bytes from one file); *a separate `@trait(...)` suffix* — new locator grammar for no discriminating power the qualified fragment doesn't already have; *crate-segment-only prefix* (qualify with just the first path segment) — `a::b::Tr` vs `a::c::Tr` still collide. **Wardline lockstep:** rides the Amendment-6 handoff (escalation-gated at rc5, batched 6–9 re-vendor). **Corpus rows landed with the implementation** (extractor-generated): `trait_path_twin`, `trait_path_twin_single_vs_multi`, `trait_path_lone_multi_segment`, `trait_path_cfg_twin_unqualified`.
**Amendment 8 (2026-06-11, `#[path]`-mounted module routing)**: Closes `clarion-bdb1eccf48`, the last Sprint-4 gold-blocker collision family, and supersedes the §1 deferral ("`#[path = "…"]` module-file overrides are deferred"). Two producers minted the same module id: the FILE WALK routed a mounted file purely by its filesystem path (`module_path_for`: tokio `src/process/unix/mod.rs` → `tokio.process.unix`, ignoring that the file is mounted as `#[cfg(unix)] #[path = "unix/mod.rs"] mod imp;`), and the AST WALK emitted the inline facade `pub(crate) mod unix { use super::imp::*; }` at the same dotted path from `process/mod.rs`; the module cfg-twin guard counted only inline mods — the file-mounted sibling is named `imp`, not `unix` — so no `@cfg` fired and `rust:module:tokio.process.unix` (and `….windows`) was emitted from two files: silent intra-run merge or a `LMWV-INFRA-PARENT-CONTAINS-MISMATCH` FailRun. **Decision — a targeted mount overlay with a filesystem default (Option A2), plus a cross-producer twin rule spanning both module-declaration forms.** At `initialize`, `mounts::discover_mounts` walks the project once (the symbol-table walk's symlink and ADR-050 guard discipline; a cheap `#[path` byte gate — whitespace-tolerant between `#`, `[`, and `path` — limits the extra syn parse to mount-bearing files), collects every `#[path = "…"] mod name;` declaration under rustc's relative-path rule (a file-level decl is relative to the declaring file's directory; a decl inside inline `mod` blocks is relative to the would-be directory of the nesting — mod-rs files anchor at their own directory, non-mod-rs files at their stem directory), and resolves each target's *logical* dotted path through a memoized fixed point: mounts chain (a mounted file's own mounts compose from its mounted path); a resolution cycle drops the cycling mounts (filesystem fallback, always terminates); a target claimed by two mounts resolves to the first by sorted (declaring-file relative path, byte offset), with the host's first-claim-wins machinery as backstop (adjudicated R5). `logical_module_path` then routes every emission — symbol table and `analyze_file` share one overlay instance, built BEFORE the table so `use`-resolution cannot desync — as: exact mount hit, else longest mounted-subtree prefix, else `module_path_for` unchanged. The subtree consequence: a `<dir>/mod.rs` target claims `<dir>/` (children rewrite onto the mount, a trailing `mod` stem collapses) and an `x.rs` target claims the exact file plus `<target_dir>/x/`. **Equal-dir subtree claims** (`#[path = "x.rs"] mod a;` + `#[path = "x/mod.rs"] mod b;` — the `x.rs` mount's child-dir claim and the `x/mod.rs` mount's dir claim are the SAME directory) resolve deterministically: children under `x/` rewrite onto the LAST-registered equal-length prefix, and registration follows the target-path `BTreeMap` order — where `x/mod.rs` sorts before `x.rs` by path components — so the **`x.rs`-target mount wins the shared directory**; the two exact target files still route by their own mounts, and R5 is untouched (it governs one target claimed by two mounts, keyed on the declaring file). **Twin rule:** module-name occurrences are counted per declaring item list across BOTH inline `mod n { … }` and decl `mod n;` forms; a twin mount appends the §3 `@cfg(...)` discriminant, and — the symmetric extract-side half — an inline mod twinned with a decl mod now takes its own suffix (`#[path = "x.rs"] #[cfg(a)] mod m;` + `#[cfg(b)] mod m { … }` → `x.rs` routes `…m@cfg(a)`, the inline emits `…m@cfg(b)`). The twin rule also **composes through inline-mod nesting**: a mount declared *inside* a cfg-twin inline mod takes that mod's `@cfg`-suffixed segment in its logical PREFIX, mirroring the AST walk's inline-mod emission byte-for-byte — `#[cfg(feature="a")] mod outer { #[path = "x.rs"] mod m; }` + `#[cfg(not(feature="a"))] mod outer { #[path = "y.rs"] mod m; }` route `x.rs` → `…outer@cfg(feature="a").m` and `y.rs` → `…outer@cfg(not(feature="a")).m` (composing the bare inline name would route both to one id — a duplicate-id family); the `#[path]` target itself still resolves against the bare would-be directory, which carries no cfg (corpus row `path_mount_inside_cfg_twin_inline_mod`). Accepted edge of the cross-form counting: the decl form counts toward twin-ness even when it never emits an entity (e.g. a symlinked module file the walk skips), so a previously-lone inline mod can gain an `@cfg` suffix from a decl sibling that produces nothing. Tokio before/after: pre-amendment `tokio.process.unix` and `tokio.process.windows` each minted TWICE (mounted file + facade, the collision); post-amendment the mounted trees route `tokio.process.imp@cfg(unix)` / `tokio.process.imp@cfg(windows)` (twin `mod imp;` decls → @cfg split) while the facades keep `tokio.process.unix` / `tokio.process.windows` — four distinct ids, and every file under a mounted directory re-keys with it (`process/unix/orphan.rs` → `tokio.process.imp@cfg(unix).orphan`, the child-prefix consequence, spelled out because it is a one-time re-key of whole subtrees). Macro-wrapped mounts (inside unexpanded macro invocations) are **invisible by dialect rule** — no producer expands macros, and the second producer must reproduce the route from one file — so the target routes by filesystem; no macro expansion is ever attempted. A `#[cfg_attr(pred, path = "…")]`-delivered mount is invisible by the same rule — only a literal `#[path]` attribute is a mount — so its target likewise falls back to filesystem routing. **Rejected:** *full rustc mod-tree resolution from the crate root* — a macro-invisible mount then orphans the whole mounted file from the resolved tree (data loss, where the overlay merely falls back to a filesystem route); *a collision-side discriminant at the duplicate-id site* — the same parent-file lookup the overlay performs, spent minting a semantically WRONG id (it must either re-parent the facade, breaking the `file_scope` contract, or delete the public facade entity outright); *cfg-suffixing the filesystem-derived name* (`tokio.process.unix@cfg(unix)`) — not rustc-true (the logical module is `imp`), re-collides as soon as the facade itself twins, and gratuitously churns mounted subtrees that never collided. **Known limitations:** the precise R6 rule is that a mount registers only when its target lies inside the **declaring crate's own `src/` root** — this rejects both what `emittable_scope` would reject (tests/, benches/, out-of-tree paths) AND a target inside a *different* crate's `src/` tree — anything else is ignored, filesystem fallback, documented; resolver: a twin mount's `@cfg` suffix won't match a bare dotted `use`-lookup, so such an import edge drops as External (the pre-existing cfg-twin resolver-mismatch class, not new to mounts); operator ergonomics: the implementation module carries its MOUNT name (`…imp@cfg(unix)`), the public facade keeps the public name (`…unix`) — readers find the facade where they expect it and the implementation under the name the source actually declares. **One-time SEI re-key:** a store analyzed pre-amendment holds mounted subtrees under their filesystem locators; the next re-analysis emits the mount-correct ids and the old bindings retire through the normal SEI absence path — no migration. **Corpus rows landed with the implementation** (extractor-generated): the `module_route` row `path_attr_known_gap` is re-pointed to pin `module_path_for`'s (unchanged) no-mount-context fallback — that harness drives `module_path_for` directly and cannot express parent-file context — and a new `module_mounts` section pins the mounted routes end-to-end (`path_mount_dir_module`, `path_mount_child_prefix`, `path_mount_chain`, `path_mount_macro_invisible_fallback`, `path_mount_inline_nested_decl` — the inline-nesting rustc-rule row). **Wardline lockstep:** module routing is sp2 for Wardline; the vendored corpus re-syncs via the escalation-gated, batched Amendments-6–9 handoff at their rc5 branch, not a push.
**Amendment 9 (2026-06-11, unnamed `const _` skip-emission)**: Closes `clarion-83870dc534`, the largest residual collision family of the Sprint-4 gold sweep — **all 15 rust-analyzer residuals** are repeated `const _: () = …;` compile-time-assertion items in one module, every pair rendering the identical `…<module>._` locator and silently merging at the writer's `ON CONFLICT(id) DO UPDATE`. (Amendments 6–8 — self-type path, trait path, `#[path]` mounts — landed 2026-06-11 as the companion paragraphs above; the whole 6–9 wave ships to Wardline as one batched re-vendor.) **Decision:** an `Item::Const` whose ident is `_` is **not an entity**. The extractor emits neither the entity, nor its `contains` edge, nor any Phase-2 `references` sites from its declared type or initializer — a finding inside one attributes to the enclosing module, the same convention as nested fns and closures. The skip is **unconditional on `ident == "_"`** — a lone anonymous const is skipped too, because skipping only when twinned would make the emitted set sibling-dependent (adding a second `const _` would delete the first one's entity) and churn SEI. The grounding is that `_` is non-identifying *by construction*: the item is un-nameable, so no path expression, `use`, finding, or association can ever target it, and no discriminant can rescue it. **Module-level only by construction:** rustc rejects associated/trait-level `const _` and `static _` entirely, `syn` rejects `static _` at parse (the existing degraded-parse path), and the extractor never walks trait bodies or assoc consts — the free-item arm is the only emission site touched. **Rejected:** an ordinal discriminant (`_@0`, `_@1`) — sibling insertion/removal churns every later twin's SEI and the resulting locators are unqueryable (nothing can name the item to ask about it); a span/line discriminant — churns on any edit above the item; a content-hash discriminant — no declaration-derived precedent in the dialect, churns on every body edit, and is not even injective for byte-identical twins (the observed corpus shape); cfg-only discrimination — proven insufficient, the rust-analyzer twins share one cfg so `@cfg` hands both the identical suffix; skip-only-when-twinned — the sibling-dependent emitted set above. **Category:** an **emitted-entity-SET amendment**, not a kind/shape change — `const` remains the kind for every *named* const, so there is **no `ontology_version` bump**. **Corpus rows landed with the implementation** (extractor-generated): `unnamed_const_skip` (a named `LIMIT` plus two `const _` twins → only the module and `LIMIT` emit; the skip is pinned as an ABSENT expected row, precedent `nested_fn_is_not_an_entity`) and `unnamed_const_cfg_twin_skip` (two same-cfg `const _` → only the module emits). **One-time SEI note:** a store analyzed before this amendment may hold an alive `rust:const:…._` chimera binding; the next re-analysis emits no entity at that locator, so the binding retires through the normal SEI absence path — no migration. **Wardline lockstep:** the vendored corpus copy must re-sync, but Wardline's Rust frontend lives on its `rc5` release branch, so the re-vendor rides the same escalation-gated handoff as Amendments 4–5; their parity gate self-enforces the skip once vendored, since `expected` arrays are complete emissions.

**Context**: The forthcoming first-party Rust language plugin (`crates/loomweave-plugin-rust`, design spec `docs/superpowers/specs/2026-06-08-rust-language-plugin-design.md`) must mint entity ids of the form `{plugin}:{kind}:{qualname}` (ADR-003). Under SEI (ADR-038) that id is the **locator** — the carry-or-mint matcher keys on it, and `sei.rs:600` enforces a partial-unique index on `current_locator WHERE status='alive'`. A multi-agent design review (2026-06-08) verified two locator-collision families in the draft qualname scheme that the writer would silently merge (`writer.rs:570` is `ON CONFLICT(id) DO UPDATE`), plus a benign-edit churn case. Because SEI is a cross-product contract (Filigree associations, Wardline taint key on it) and the scheme has no Python precedent, it is settled here before any code lands.
**Relates to**: [ADR-003](./ADR-003-entity-id-scheme.md) (the `{plugin}:{kind}:{qualname}` scheme this canonicalizes for Rust), [ADR-022](./ADR-022-core-plugin-ontology.md) (`rule_id_prefix`, plugin-owned ontology), [ADR-028](./ADR-028-edge-confidence-tiers.md) (the confidence ladder the syntactic backend is honest about), [ADR-038](./ADR-038-sei-token-and-signature.md) (the qualname is the SEI *locator*; this ADR keeps it collision-free within a run), [ADR-002](./ADR-002-plugin-transport-json-rpc.md) (JSON-RPC transport only).

## Summary

Two decisions:

1. **Qualname canonicalization (the load-bearing one).** A Rust entity's `canonical_qualified_name` is a **dot-separated, crate-rooted, impl-discriminated path** that is (a) **unique within a run** — no two distinct source items collapse to one locator — and (b) **stable across benign edits** — reordering items, adding/removing a sibling, or renaming a generic type parameter does not churn an unrelated entity's id. This directly closes the cross-crate collision (no crate token) and the intra-type collisions (missing method/impl discriminator) the review found.

2. **Syntactic backend (`syn`), recorded.** The plugin parses with `syn`, not rust-analyzer. The binding reason is the project-wide **network-free / credential-free** posture (CLAUDE.md "Local-first"): `cargo metadata` / registry resolution is network egress, and a buildable-project requirement pulls a toolchain into `analyze`. rust-analyzer is admissible **later** only as strictly-additive *edge-confidence* enrichment (ADR-028 ladder), and only if it reproduces this ADR's qualname byte-for-byte.

## Context

### The locator must be collision-free within a run

Verified against source on 2026-06-08:

- `crates/loomweave-storage/src/writer.rs:570` upserts entities `ON CONFLICT(id) DO UPDATE SET …` — a second entity with the same id **overwrites** the first's byte range, content hash, and parent. There is no uniqueness guard at id assembly: `crates/loomweave-core/src/entity_id.rs` validates the `[a-z][a-z0-9_]*` grammar for `plugin_id`/`kind` only and rejects a literal `:` in the qualname (line 141) — nothing else. So a colliding qualname is *silent intra-run data loss*, not an error.
- `crates/loomweave-storage/src/sei.rs:600` enforces a partial unique index on `current_locator WHERE status='alive'`. The SEI matcher (ADR-038) carries-or-mints by reading the prior binding **keyed on the locator**; two alive entities sharing a locator break that index and confuse the matcher.

The draft scheme (spec §4.1/§4.2 as first written) collided in idiomatic Rust:

| Colliding source | Draft locator (both) | Why it collides |
|---|---|---|
| `loomweave_core::config::X` + `loomweave_cli::config::X` | `rust:struct:config.X` | no crate token (8-crate dogfood target) |
| `impl Display for Foo { fn fmt }` + `impl Debug for Foo { fn fmt }` | `rust:function:…Foo.fmt` | method qualname had no impl/trait discriminator |
| `impl From<i32> for Foo` + `impl From<u32> for Foo` | impl: `…Foo.impl[From]` | trait's generic arg omitted from key |
| two inherent `impl Foo {}` blocks | `…Foo.impl#` | empty generic sig, no ordinal |
| `#[cfg(unix)] fn f` + `#[cfg(windows)] fn f` | `…mod.f` | §5 all-cfg-visible policy makes this *guaranteed* |

And one benign-edit **churn** case: the generic type-parameter *name* was in the key (`impl#<T>`), so renaming `<T>`→`<U>` — a no-op edit — churned the id, orphaning bindings.

### What rust-analyzer would cost the analyze path

CLAUDE.md ("Local-first"): the only required network egress is the LLM provider during `summary`; `analyze` runs with no credentials. rust-analyzer needs Cargo metadata (registry/index resolution = network) and a buildable project (toolchain). That is the binding exclusion. RAM/index cost is real but secondary. ("Build-free" is this design's own derived rule, not an inherited CLAUDE.md term — the inherited terms are network-free and credential-free.)

## Decision

### 1. Qualname canonicalization

A Rust entity's `canonical_qualified_name` is `<crate>.<module-path>.<item-path>`, dot-separated, where:

**Crate token (closes B1).** The leading segment is the crate name (underscored, e.g. `loomweave_core`). Crate roots are discovered by reading each `Cargo.toml`'s `[package].name` **as text** within `project_root` (permitted — the hard constraint forbids running `cargo metadata`/registry resolution, not reading a manifest file), falling back to the directory containing `src/lib.rs` / `src/main.rs`. A virtual workspace with no member resolves crate-by-nearest-manifest.

**Module path.** The `mod` tree from crate root to the item, following file-module (`mod foo;` → `foo.rs`/`foo/mod.rs`) boundaries. Inline `mod` blocks nest normally. **`#[path = "…"]`-mounted modules route to their MOUNTED logical path** (Amendment 8, superseding the original deferral): a targeted mount overlay — exact mounted file plus mounted-subtree prefixes, twin mounts split by `@cfg` — with the pure filesystem derivation as the default for every unmounted file. Macro-wrapped mounts are invisible (filesystem fallback); out-of-src targets are ignored.

**Item path with impl discrimination (closes B2).**

- Free items: `<crate>.<mods>.<name>` (function, struct, enum, trait, type_alias, const, static, macro). An unnamed **`const _`** is skipped entirely — no entity, no `contains` edge, no `references` sites (Amendment 9): `_` is non-identifying and un-nameable, so the item has no locator and a finding inside one attributes to the enclosing module.
- `impl` blocks get a synthesized, **genuinely source-order-independent** discriminator appended to the target type, and the impl block is its own `impl` **entity** (its methods parent to it, ADR-026 `module -> impl -> method` containment):
  - **Self-type concrete generic arguments are part of `<Type>`** (amended 2026-06-09, self-type-args). `<Type>` in every impl locator carries the self type's concrete generic arguments: `impl Foo<i32>` → `…Foo<i32>`, `impl Foo<u32>` → `…Foo<u32>`. Without this, both render bare `…Foo`, so their impl keys (`Foo.impl#<>` / `Foo.impl[Display]`) collide; the extraction layer's `seen_impl_ids` then **spuriously merges** the second impl into the first, re-parents both impls' methods under one id, and a like-named method (e.g. two `get`) is **silently overwritten** at the writer's `ON CONFLICT(id) DO UPDATE` — silent intra-run data loss. A **top-level** self-type arg that is *directly* the impl's **own declared generic parameter** is rendered **positionally** (`$0`, `$1`, … matching the inherent `#<…>` De Bruijn scheme), so `impl<T> Foo<T>` → `…Foo<$0>` and a `<T>`→`<U>` rename does not churn; **concrete** args (`i32`, `String`, a const) render whitespace-free via the shared `type_textual` helper **and through the `escape_reserved(strip_ws(arg))` pipeline (amended 2026-06-10): a path-typed arg escapes its `::` (`Foo<std%3A%3Aio%3A%3AError>`), and const args get the same `strip_ws` as type args (`Foo<{1+2}>`, not the proc-macro2-spaced form)**; lifetimes and associated-type bindings are dropped (not part of the locator). **Known limitation (rename-stability is top-level only):** a declared param nested *inside* another type argument — `impl<T> Foo<Vec<T>>`, `impl<T> Foo<&T>` — is NOT positionally substituted; the whole arg renders via `type_textual` (`Foo<Vec<T>>`, `Foo<&T>`), so a rename of a nested param DOES churn the locator. This is strictly better than the pre-amendment behaviour (which silently *dropped* a method) but is not full rename-stability; recursive positional rendering (`Foo<Vec<$0>>`) is a deferred follow-up. A second-producer (Wardline) MUST match this `type_textual` rendering of nested params, not implement recursive positional substitution; this is pinned byte-for-byte by the `generic_self_nested_param` corpus row (`impl<T> Foo<Vec<T>>` → `…Foo<Vec<T>>`, NOT `…Foo<Vec<$0>>`), so a recursive-positional second producer trips the conformance gate rather than diverging silently. A non-generic self type (`impl Foo`) renders bare `Foo`. The full impl locator composes in a fixed order: `<self-type-with-args>.<impl#<sig>|impl[Trait]>[@cfg(...)]` (self-type args first, then the inherent/trait discriminant, then any cfg suffix). This composes cleanly with the inherent positional signature and the `@cfg(...)` suffix below; `<`, `>`, `$` are already legal in ids (`entity_id.rs` reserves only `:`). The `<Type>` **base** is the self type's last path segment in the common case; when a post-cfg impl-qualname group carries ≥ 2 distinct *written* self-type paths, every `Type::Path` member's base qualifies to its escaped written path (`a%3A%3AX`) — **Amendment 6**, stage S of the residual-collision ladder.
  - **Trait impl:** `…<Type>.impl[<TraitPath-with-concrete-generics>]` — the trait's concrete generic arguments are **part of the key** (`impl[From<i32>]`, `impl[From<u32>]` are distinct), rendered through the same `escape_reserved(strip_ws(arg))` pipeline (amended 2026-06-10: `impl[From<std%3A%3Aio%3A%3AError>]`). With the self-type-args amendment, `impl Display for Foo<i32>` → `…Foo<i32>.impl[Display]` and `impl Display for Foo<u32>` → `…Foo<u32>.impl[Display]` are distinct even though the trait fragment is identical. The trait fragment keys on the path's **last segment** in the common case; when a post-S impl-qualname group carries ≥ 2 distinct *written* trait paths, every member's fragment switches to the escaped written path (`impl[tokio%3A%3Aio%3A%3AAsyncRead]`) — **Amendment 7**, stage T of the residual-collision ladder.
  - **Inherent impl:** `…<Type>.impl#<positional-generic-signature>`, with a **positional, De Bruijn-style** rendering of generic parameters (each param rendered as `$0`, `$1`, … so a param *rename* does not churn: `impl<T> Foo<T>` and `impl<U> Foo<U>` both render `impl#<$0>`; a non-generic `impl Foo` renders `impl#<>`). There is **no source-order ordinal** (amended 2026-06-09, Option (b)): multiple inherent impls of the same `(self-type-with-concrete-args, positional-generic-signature, cfg)` **merge** into one `impl` entity (their union of methods all hang off that single entity), making the discriminator both *reorder-stable* and *method-set-stable*. The merge key now includes the concrete self-type args (self-type-args amendment), so two blocks on the **same** instantiation (`impl Foo<i32> { … } impl Foo<i32> { … }`) still merge, while two blocks on **different** instantiations (`impl Foo<i32>` vs `impl Foo<u32>`) do **not**. Two inherent blocks for the same type in different files stay distinct because the defining file's module path is already in the prefix. This removes the original draft's self-contradiction (an ordinal "assigned by source order" cannot be "source-order-independent"); merging is the resolution.
  - **cfg-twin impls** (same `(type, generic-sig)`, mutually-exclusive `#[cfg]`s — applicable to trait impls too, e.g. `#[cfg(unix)] impl Display for Foo` / `#[cfg(windows)] impl Display for Foo`) are split by the canonicalized `@cfg(<predicate>)` suffix on the impl key (`impl#<>@cfg(unix)` vs `impl#<>@cfg(windows)`), so they do **not** merge; their methods inherit the suffix. This is the only thing keeping cfg-twin impls (and their methods) distinct once the ordinal is gone. **cfg-twin *methods*** — two cfg-gated methods that land on the *same* impl entity (one block, or several merged under Option (b)), which the impl-level suffix cannot tell apart within one entity — are split by a method-level `@cfg(...)` suffix appended *after the method name* (`…impl#<>.go@cfg(unix)`; amended 2026-06-11, `clarion-dfeb905f46`), keyed on the final impl qualname + method name so an already-split cfg-twin impl gets no redundant method suffix.
  - The `[`, `]`, `#`, `<`, `>` characters are permitted by `entity_id.rs` (only `:` is rejected). A fixture row MUST exercise each so downstream consumers (Wardline expr parser, Filigree glob, guidance `match_rules`) are audited against them (review finding L5).
- **Methods carry their impl's discriminator:** a method is `…<Type>.impl[<Trait>].<method>` / `…<Type>.impl#<sig>.<method>`, never the bare `…<Type>.<method>`. This is what makes `Display::fmt` and `Debug::fmt` on the same type distinct.

**`#[cfg]` collision policy (closes the guaranteed B2 case).** Because §5 of the spec keeps *all* cfg variants visible, two same-path items gated on mutually-exclusive cfgs would collide. Each cfg-gated item that shares a path with a sibling appends a canonicalized `@cfg(<predicate>)` discriminant derived from its `#[cfg(...)]` attribute (normalized: sorted, whitespace-stripped). An item with no cfg, or a unique path, gets no `@cfg` suffix — so the common case is unchanged and only genuine cfg-twins pay the discriminator. This applies to module-level free items, impl blocks, **and impl-member methods** (`ImplItem::Fn` — amended 2026-06-11): a method gated on a `#[cfg]` that shares its `(impl-entity, name)` with a twin appends the suffix after the method name.

**Uniqueness is an invariant, not a hope.** The plugin asserts, over its emitted set per run, that **no two entities share a locator**. This is enforced by a dedicated uniqueness test class (below), not left to the writer's silent merge.

### 2. Syntactic backend

`syn` (full-fidelity typed AST, `proc-macro2` spans). rust-analyzer is excluded from the `analyze` path for the network/credential reason above and admitted only as a future, gated, strictly-additive Phase-3 *edge-confidence* enrichment that MUST reproduce §1's qualname byte-for-byte (any newly-revealed macro entity goes through this same locator + SEI + parity-fixture contract — it is *not* free additivity; review finding H4).

## Consequences

- **Parity fixture.** `fixtures/entity_id.json` is extended with Rust rows exercising the crate token, trait-impl-method, inherent-impl-method (no ordinal — amended), the inherent `impl` entity locator, positional-generic stability, a cfg-twin inherent-impl method (distinct by the `@cfg` suffix on the impl key, which replaces the dropped ordinal), and a `@cfg` free-item twin — so the byte-parity gate is non-vacuous (review finding H1).
- **Test classes.** Two dedicated test files: `identity_stability.rs` (reorder impls / mutate method-set / rename a generic param → no *other* id changes, and the renamed-generic id is itself unchanged) and `identity_uniqueness.rs` (a corpus containing every collision pair in the Context table asserts zero duplicate locators).
- **Init-time symbol table.** The crate-root + `mod`-tree discovery this ADR requires is the same walk the spec's §2.3 project symbol table performs at `initialize` (`InitializeParams::project_root`, `protocol.rs:328`); the two share one traversal.
- **SEI.** The locator stays stable across the benign edits ADR-038's matcher must tolerate (reorder, sibling add/remove, generic-param rename), so carry-not-mint holds for unaffected entities; no `entities`/`sei_bindings` shape change is implied.
- **No core change required for assembly.** The plugin calls `loomweave_core::entity_id()`; this ADR constrains only the qualname *string* the plugin hands it. The grammar/colon rules in `entity_id.rs` are unchanged.
- **Supersedes nothing.** ADR-003's scheme stands; this is the Rust-specific canonicalization of its opaque qualname segment, mirroring how the Python plugin owns its dotted qualnames.

## Weft vocabulary verdict (per ADR index acceptance rule)

No new cross-product-visible term is introduced. "locator", "SEI", and the `{plugin}:{kind}:{qualname}` shape are all pre-registered (ADR-003, ADR-038, `docs/suite/glossary.md`). The Rust qualname canonicalization is plugin-internal detail behind the opaque locator — consumers MUST NOT parse it (the opacity discipline of ADR-003/ADR-038). Verdict: **`no clash`** — nothing new to register.
