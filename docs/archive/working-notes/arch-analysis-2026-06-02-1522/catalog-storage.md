## clarion-storage (delta analysis: 2026-06-02 vs prior 2026-05-22)

**Location:** `crates/clarion-storage/`

**Responsibility:** Sole SQLite owner for the per-project Clarion database. Applies PRAGMA discipline, runs schema migrations, routes all mutations through a single writer-actor task, hands out short-lived read-only connections from a deadpool-sqlite pool, enforces wire-level graph invariants at the writer boundary, and provides the full read-side query catalogue used by MCP tools and the HTTP read API. Since the prior analysis, the crate has grown three new subsystems — Stable Entity Identity (SEI), prior-index retention, and Wardline taint-fact storage — and has closed both previously flagged schema-identity gaps (`application_id` and `user_version`).

**Patterns Observed:**

- **Actor + bounded mpsc + oneshot-ack-per-command** — unchanged. 18-variant match in `run_actor` (`writer.rs:162-319`).
- **Query-time writes route through `query_time_write` wrapper** (`writer.rs:855-877`) — all 6 new SEI/prior-index/wardline commands correctly use this path, committing any open run-batch first and re-opening it after.
- **`blake3` token minting with 0x00 separator** (`sei.rs:53-60`) — the null-byte prevents `(ab, c)` aliasing `(a, bc)`. The fixed 32-hex-char truncation gives 128 bits of identity. Tests verify the separator property (`sei.rs:703-706`).
- **Fail-closed SEI matching** (`sei.rs:278-335`) — only two confident proofs trigger a carry (git rename + unchanged body; or exactly-one-vanished identical body+sig); any ambiguity mints fresh. Null body or null signature abstains from the move predicate (`sei.rs:310`).
- **Partial unique index for alive locator** (`0005_sei.sql:46-48`): `ux_sei_alive_locator WHERE status = 'alive' AND current_locator IS NOT NULL` — orphaned/superseded bindings retain their former locator for audit without blocking a new entity from claiming it.
- **Shape-independent prior index** (`prior_index.rs:1-18`, `0004_sei_prior_index.sql`) — deliberately carries no `sei` column so it can ship before SEI lock. `schema.rs` tests assert this at build time (`schema.rs:360-375`).
- **Chunked `IN(...)` at 500** for batch reads (`wardline_taint.rs:166, 215, 273`) — avoids SQLite's variable-count limit in all production batch sizes.
- **`application_id` enforcement at write-open time** (`pragma.rs:59, 83-108`) — 0 → upgrade in place; `CLARION_APPLICATION_ID` → pass; other → `ForeignDatabase` error.
- **`user_version` forward-incompatibility check** (`schema.rs:109-127`) — called in both `apply_migrations` and `Writer::spawn` (`writer.rs:90-94`); a future-built DB is refused before any SQL runs.
- **Compile-time migration-count assertion** (`schema.rs:55-63`) — `CURRENT_SCHEMA_VERSION` must equal `MIGRATIONS[last].version`; mismatch is a compile error.
- **`RESERVED_ENTITY_KINDS` facade-bypass** (`writer.rs:537`) — direct import of `clarion_core::plugin::manifest::RESERVED_ENTITY_KINDS` couples storage to a plugin-manifest implementation detail rather than a storage-stable constant. Present since the prior analysis; not resolved.

2. **`user_version` gap — RESOLVED.** Prior analysis: "No `user_version`." 3. **`blake3` vestigial dep — RESOLVED.** Prior analysis: "declared dep but unused inside `src/`." Current: actively used in `sei.rs:54` (token minting) and `query.rs:433` (live whole-file hash). No longer vestigial.

4. **Schema table count vs. detailed-design.** `detailed-design.md` §Schema (outline, lines 615-796) documents 7 concrete tables + 1 FTS5 virtual table + 1 view, all in migration 0001. Actual migrations now create 13 tables + 1 FTS5 table + 1 view across 6 migration files. The 6 additions (`wardline_taint_facts`, `sei_prior_index`, `sei_bindings`, `sei_lineage`, `schema_migrations` itself, plus the `briefing_blocked` ALTER on `entities`) are completely absent from detailed-design.md's schema section. Severity: **Medium** — no correctness risk (migrations are the truth), but a maintainer reading only detailed-design.md would miss 6 tables and the `entities.signature` column added by migration 0005.

5. **`entities` column drift vs. detailed-design.** `detailed-design.md:615-642` shows `entities` with columns `summary TEXT, wardline TEXT`. Current migration 0001 names them `summary_json` and `wardline_json` (verified in `commands.rs:66,68`). Severity: **Low** — detailed-design is an outline, not a normative source; migrations are the source of truth per the CLAUDE.md precedence hierarchy.

6. **`detailed-design.md` concurrency section drift.** The doc describes "checkpoint issued after each 10 analyze-transactions" and a `clarion analyze --shadow-db` flag. Neither feature is present in the current codebase. `wal_autocheckpoint=1000` (not 10) is the actual cadence; no `--shadow-db` flag exists. Severity: **Low** — the doc is an "outline" baseline, but the quantitative claim (10 vs. 1000) could mislead operational tuning.

7. **`RESERVED_ENTITY_KINDS` facade-bypass — STILL PRESENT.** `writer.rs:537` imports `clarion_core::plugin::manifest::RESERVED_ENTITY_KINDS` directly. The prior analysis flagged this as a coupling leak (storage reaches into a plugin-manifest implementation detail rather than a storage-stable constant). Unchanged. Severity: **Low** — functional, but any refactor of the plugin-manifest module path breaks the storage crate's build.

8. **`detailed-design.md` transaction cadence.** Doc says "N files per transaction (default N=50)." Code uses N writes per batch (entities + edges + findings each count), not N files. `DEFAULT_BATCH_SIZE = 50` at `writer.rs:34`. The distinction matters for scale estimates: a file with 1000 functions produces 1000+ writes, not 1. Severity: **Low** — operational, not correctness.

**Quality Concerns:**

- **`WriterCmd` enum: 18 variants in one match arm, writer.rs is 1211 LOC.** Prior concern was 11 variants / 1074 LOC. The file has grown by 137 LOC. The `run_actor` match (`writer.rs:162-319`) is now 157 lines of dispatch. Each new SEI/prior-index command requires three edit sites (enum variant, match arm, handler function). The pattern remains clean but is crossing "should split" threshold. Severity: **Low** — maintainability only.

- **`query.rs` growth: 1727 LOC (was ~1161).** Added ~566 LOC of new query helpers since prior analysis, all hand-written SQL strings. No prepared-statement caching. 1:1 LOC ratio with tests remains. Severity: **Low/Medium** — consistent with the existing pattern; risk of SQL string drift between similar helpers increases with size.

- **SEI matcher reads all alive bindings into a `HashMap<String, SeiBinding>` at the start of a re-index.** `alive_bindings_snapshot` (`sei.rs:231-246`) loads the full alive set. At 500k entities this is a ~500k-entry HashMap held in memory for the duration of the analyze pass. No streaming or incremental read. Severity: **Medium** — no correctness risk; memory footprint concern at `elspeth`-scale. The doc (`prior_index.rs:9`) notes "called once at the start of a re-index" as a known cost.

- **`get_taint_facts_by_sei` dedup is done in Rust, not SQL** (`wardline_taint.rs:212-255`). The function builds a `HashMap<sei, TaintFactRow>` post-query because "rows per SEI is tiny." If a caller accidentally writes duplicate explicit SEIs, the ORDER BY + first-wins logic in Rust is correct but allocates N rows before deduplication. At batch sizes ≤500 this is not a practical concern. Severity: **Low**.

- **No test for `WriterCmd::ResumeRun`** evident in the grep scan of writer.rs tests. The resume path (`writer.rs:428-445`) updates an existing run row back to `running`, clears `completed_at`, and re-opens the transaction. A missing test for this path means a regression (e.g. failing to clear `completed_at`) would surface only in e2e tests. Severity: **Medium** — specific regression risk, not just coverage percentage.

- **`orphaned_bindings` sort (`sei.rs:357-359`) is `sort_unstable`** — documented as "deterministic order (map iteration is not stable)." Correct for a pure-value slice of SEI strings. No concern.

- **`TaintFact` struct has no `sei` field in the inline test helpers** (`wardline_taint.rs:406-422`): the test constructs `TaintFact { ..., sei: ... }` but the `sei` field is not shown — verified the struct definition at `wardline_taint.rs:90-106` does include `pub sei: Option<String>`. The upsert test (`wardline_taint.rs:401`) uses the struct without the field (Rust fill-in pattern), meaning the SEI-stamping column goes untested in the basic upsert round-trip. Severity: **Low** — a dedicated `get_taint_facts_by_sei` write test would cover this path.

- **`replace_prior_index` uses `conn.unchecked_transaction()` (`prior_index.rs:188`)** — opens a transaction on a connection that may already have a transaction open. The doc comment at `prior_index.rs:183-195` notes "Runs on the writer-actor connection in autocommit context (the `query_time_write` path commits any open run batch first)." This is correct given the call sequence, but the "unchecked" is a footgun if called outside that path. Severity: **Low** — by-convention safety, not enforced.
