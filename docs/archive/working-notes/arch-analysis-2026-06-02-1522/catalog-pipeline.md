## Analysis Pipeline (`clarion analyze`)

**Location:** `crates/clarion-cli/src/analyze.rs`, `clustering.rs`, `sei_git.rs`, `analyze_lock.rs`, `run_lifecycle.rs`

**Responsibility:** Orchestrates the full `clarion analyze` execution: discovers plugins, walks the source tree, applies pre-ingest secret scanning, fans out per-plugin structural extraction (Pattern A buffering), ingests entities/edges through the writer-actor in FK-safe order, runs Leiden clustering to materialise subsystems, runs the Wave 1 SEI mint/carry pass to assign stable entity identifiers, emits findings to Filigree (Phase 8/8b), and closes the run row with a correct terminal status (Completed / SoftFailed / HardFailed). Cross-process concurrent analysis is blocked by an `fs2` advisory lock.

**Key Components:**

- `analyze.rs:206` ‚Äî `run_with_options` (~836 LOC, `#[allow(clippy::too_many_lines)]`): the single async function containing the entire pipeline. Despite extractive helpers, this remains a monolith ‚Äî see Quality Concerns.
- `analyze.rs:166` ‚Äî `AnalyzeOptions`: carries all pipeline flags including `no_sei`, `no_incremental`, `resume_run_id`, `prune_unseen`, `progress_file`, `legis_url`.
- `analyze.rs:58` ‚Äî `ProgressReporter`: atomic best-effort JSON snapshot writer; supports MCP `analyze_status`; no-op on CLI path.
- `analyze.rs:1085` ‚Äî `run_sei_mint_pass` (~192 LOC, `#[allow(clippy::too_many_lines)]`): Wave 1 SEI identity pass ‚Äî reads prior alive bindings, consults `sei_git::select_git_rename_source` for rename hints, calls `rebind_or_mint` per entity, orphans vanished bindings, writes `UpsertSeiBinding` + `AppendSeiLineage` + `SetEntitySignature`; runs after `CommitRun` (Completed only), enrich-only and skippable via `--no-sei`.
- `analyze.rs:1307` ‚Äî `run_phase3_clustering` (~231 LOC, `#[allow(clippy::too_many_lines)]`): Phase 3 subsystem materialisation ‚Äî issues `FlushRunBatch`, opens a direct `rusqlite::Connection`, loads module entity IDs + dependency edges, runs `cluster_modules`, emits `core:subsystem:<hash>` entities + `in_subsystem` edges, emits `CLA-FACT-CLUSTERING-WEAK-MODULARITY` finding when `modularity_score < threshold`.
- `analyze.rs:1680` ‚Äî `emit_findings_to_filigree`: Phase 8 Filigree emission (WP9-B, REQ-FINDING-03) ‚Äî best-effort POST to `/api/v1/scan-results`; gated on `integrations.filigree.emit_findings`; uses a plain OS thread to avoid `reqwest::blocking` runtime conflict.
- `analyze.rs:1848` ‚Äî `prune_unseen_findings_in_filigree`: Phase 8b retention sweep (`--prune-unseen`, REQ-FINDING-06) ‚Äî POSTs `CleanStaleRequest` to Filigree; same OS-thread pattern; enrich-only.
- `analyze.rs:2155` ‚Äî `run_plugin_blocking` (~155 LOC, `#[allow(clippy::too_many_lines)]`): synchronous worker run inside `spawn_blocking` ‚Äî spawns plugin via `PluginHost`, iterates files calling `analyze_file`, collects entities/edges/signatures into memory, reaps child, classifies SIGKILL/SIGSEGV as OOM findings.
- `clustering.rs:52` ‚Äî `cluster_modules` / `cluster_modules_with_algorithms`: Leiden via `xgraph`, falls back to `local_weighted_components` when Leiden returns ‚â§1 community; deterministic via configurable seed; `cluster_hash` = SHA-256 of sorted member IDs, truncated to 12 hex chars.
- `sei_git.rs:61` ‚Äî `ShellGitRenameSource`: shells `git diff --name-status -M HEAD`, translates file renames to locator renames over the current run's locator set; Python-shaped `path_to_module` (strips `.py`/`.pyi`, handles `__init__`).
- `sei_git.rs:196` ‚Äî `LegisGitRenameSource` (WS9): reads `legis`'s `GET /git/renames?rev_range=‚Ä¶`; activated only for committed rev-ranges when `legis` is configured and reachable; 3 s `LEGIS_HTTP_TIMEOUT`.
- `sei_git.rs:310` ‚Äî `select_git_rename_source`: capability-aware selector ‚Äî shell source for working-tree window (default); Legis only for committed base + configured + reachable; short-circuits network probe on empty base.
- `analyze_lock.rs:53` ‚Äî `acquire_analyze_lock`: `fs2` exclusive lock on `.clarion/clarion.lock`; RAII guard (`#[must_use]`) must outlive the writer-actor `JoinHandle::await` to prevent WAL races.
- `run_lifecycle.rs:4` ‚Äî `begin_run` / `resume_run` / `open_run`: thin writer-actor wrappers for `BeginRun` / `ResumeRun`; `open_run` centralises the `--resume` dispatch so all early-exit paths (discovery error, no-plugins) honour it uniformly.

**Dependencies:**

- Inbound: `crates/clarion-cli/src/main.rs:73` (`rt.block_on(analyze::run_with_options(‚Ä¶))`) ‚Äî CLI path only. MCP `analyze_start` (`clarion-mcp/src/lib.rs:1963`, `analyze_runs.rs:51`) spawns a fresh `clarion analyze` subprocess via `spawn_analyze`, so MCP is an inbound OS-level caller, not a Rust caller.
- Outbound (Clarion crates): `clarion-core` (`PluginHost`, `CrashLoopBreaker`, `discover`, entity-ID assembler, `HostError`, `HostFinding`, `FINDING_DISABLED_CRASH_LOOP`); `clarion-storage` (`Writer`, `WriterCmd`, `alive_bindings_snapshot`, `mint_sei`, `rebind_or_mint`, `orphaned_bindings`, `module_dependency_edges`, `PriorIndexEntry`, `SeiBindingRecord`, `SeiLineageEntry`, `GitRename`, `GitRenameSource`, schema/pragma helpers); `clarion-mcp` (`McpConfig`, `FiligreeHttpClient`, `scan_results::*`, `filigree_url::resolve_filigree_url`).
- Outbound (external): `rusqlite::Connection` (direct read connection, bypassing reader pool ‚Äî two sites: incremental skip at `:413`, phase 3 at `run_phase3_clustering:1343`); `fs2` (advisory lock); `xgraph` (Leiden); `ignore::WalkBuilder` (tree walk); `reqwest::blocking` (Legis HTTP probe; Filigree emission run on plain OS threads); `sha2`/`hex` (cluster hash, file hash).
- Outbound (processes): `clarion-plugin-*` executables via `PluginHost::spawn`; `git` binary via `ShellGitRenameSource::run_git_diff`; `legis` HTTP endpoint (optional, enrich-only).
- Outbound (storage): `.clarion/clarion.db` (writer-actor + two direct read connections per run); `.clarion/clarion.lock` (advisory lock file).

**Patterns Observed:**

- **fs2 advisory cross-process lock** (`analyze_lock.rs:53`): `acquire_analyze_lock` acquires an exclusive lock on `.clarion/clarion.lock` before writer-actor spawn; RAII guard drop-order is load-bearing (must outlive `handle.await` at `analyze.rs:1024`).
- **Orphan-run recovery before writer spawn** (`run_lifecycle.rs:4`, called at `analyze.rs:388`): `open_run` uses `BeginRun`/`ResumeRun` via the just-spawned writer-actor; prior `recover_preexisting_running_runs` (now removed from run_lifecycle.rs and inlined into writer init or startup) is no longer a separate step ‚Äî verified: `run_lifecycle.rs` now has only `begin_run`/`resume_run`/`open_run`.
- **Secret-scan before BeginRun** (`analyze.rs:387‚Äì388`): `pre_ingest` completes *before* `open_run`, so no plugin ever receives `.env` contents; findings attach to entity anchors from plugin output *after* ingestion (`persist_findings` at `:749`).
- **Two crash-loop breakers** ‚Äî the `CrashLoopBreaker` instance (`analyze.rs:468`) guards the per-plugin loop; `PluginHost`'s internal breaker guards the per-file loop inside `run_plugin_blocking`. Plugin-task panic is normalised into a crash reason by `handle_plugin_task_join_result` (`analyze.rs:2053`) so a blocking-task panic never bypasses the CommitRun/FailRun block.
- **FK-ordered ingestion** (`analyze.rs:645‚Äì724`): `InsertEntity` before `ReplaceUnresolvedCallSitesForCaller` before `InsertEdge`; comment at `:640` names this load-bearing.
- **Deterministic clustering seed** (`clustering.rs:74‚Äì131`): `CommunityConfig { deterministic: true, seed }` passed to `leiden_clustering`; seed is config-recorded in run stats.
- **Wave 1 SEI mint / carry-forward** (`run_sei_mint_pass:1085`): runs post-`CommitRun(Completed)`, enrich-only ‚Äî a failure logs and is swallowed (`analyze.rs:928‚Äì934`). `--no-sei` skips entirely (`analyze.rs:908`). Orphan writes precede carry/mint writes to avoid transient alive-locator unique-index double-binding (T2.2 Step 5, `:1169‚Äì1200`).
- **Wave 2 incremental skip** (`analyze.rs:411‚Äì541`): partitions plugin files into re-analyse vs skip based on whole-file hash comparison; skipped files' locators are injected into `retained_locators` so the SEI matcher never falsely orphans still-present entities; secret-finding files are excluded from skip.
- **Wave 0 prior-index flush** (`analyze.rs:950‚Äì963`): `UpsertPriorIndex` runs after `CommitRun(Completed)` only; SoftFailed/HardFailed runs deliberately leave the snapshot at the last fully-successful run.
- **Phase 8/8b Filigree emission on OS threads** (`analyze.rs:1745‚Äì1753`, `:1885‚Äì1893`): `reqwest::blocking` client built/run/dropped on a plain OS thread inside `spawn_blocking` to avoid the "nested runtime" drop panic that occurs when a blocking reqwest runtime is dropped on a tokio worker.
- **`ProgressReporter` atomic temp-rename writes** (`analyze.rs:146‚Äì159`): writes go to a `.json.tmp` sidecar and are `rename`d into place; failed writes are logged at DEBUG and dropped ‚Äî progress is advisory, never run-fatal.

**DRIFT ‚Äî Code vs ¬ß6 Phase Flow / Determinism / Failure Model / Detailed-Design Phase-7 Rules:**

The design doc (system-design.md ¬ß6) is a **lagging document** relative to the code. Per CLAUDE.md precedence (ADR > system-design), the design doc is the bug where they diverge. Each item names the doc claim, the code reality, and the governing ADR or WP reference.

1. **Phase list mismatch ‚Äî doc ¬ß6 is incomplete (HIGH).** ¬ß6:579‚Äì603 documents nine phases (0‚Äì8) with Phase 0 (budget/cost), Phases 4‚Äì6 (LLM haiku/sonnet/opus), Phase 7 (Wardline ingest, structural findings, entity-set diff), Phase 8 (emit catalog JSON + stats). The code in `run_with_options` implements: structural extraction (‚âàPhase 1 + partial 1.5), secret scan, per-plugin orchestration, phase 3 clustering, Phase 8/8b Filigree emission ‚Äî plus three *uncharted* Wave passes: Wave 2 incremental skip (`:411`), Wave 0 prior-index (`UpsertPriorIndex:950`), Wave 1 SEI mint (`run_sei_mint_pass:1085`). Phases 0, 2, 4‚Äì7 are absent entirely. ¬ß6 carries no deferral notice for LLM phases (unlike ¬ß8's explicit "deferred to v1.1" box at line 773 and ¬ß9's at line 988). ADR-030 governs on-demand summary; no ADR explicitly supersedes ¬ß6's phase diagram.

2. **Phase 7 structural findings not implemented (HIGH drift; doc vs code).** detailed-design.md ¬ß1040‚Äì1052 and system-design.md ¬ß6:650‚Äì658 specify three phase-7 rules: `CLA-FACT-TIER-SUBSYSTEM-MIXING`, `CLA-FACT-SUBSYSTEM-TIER-UNANIMOUS`, `CLA-FACT-ENTITY-DELETED`. Repo-wide `grep` across `crates/` finds zero occurrences of any of these rule IDs. The entity-set-diff mechanism described in ¬ß6:639‚Äì648 (compare entity IDs against prior `stats.json`, emit `CLA-FACT-ENTITY-DELETED` against `core:deleted:{id}`) is also absent. Entity vanishing is instead tracked via SEI `orphaned_bindings` (`:1165`), which records lineage but emits no finding. Phase 7 as a discrete stage does not exist in the code.

3. **Deletion detection mechanism replaced (HIGH drift).** Doc (detailed-design.md ¬ß1066‚Äì1075): entity-set diff at Phase-7 boundary, reading prior `stats.json`, emitting `CLA-FACT-ENTITY-DELETED` + `CLA-FACT-GUIDANCE-ORPHAN`, invalidating summary cache. Code: vanished entities are instead detected by `orphaned_bindings` in `run_sei_mint_pass` (`:1165`), which writes `LineageEvent::Orphaned` records. No finding is emitted; no guidance-orphan check; no cache invalidation is present in the pipeline. The mechanisms are architecturally disjoint.

4. **Determinism caveat not in doc (MEDIUM drift).** ¬ß6:619 claims "byte-identical entity/edge/finding state" on back-to-back runs. `run_sei_mint_pass` explicitly carves SEI values out: "SEI *values* are not part of the byte-identical-run guarantee ‚Äî two from-scratch runs mint different SEIs" (`:1079‚Äì1083`). The guarantee holds for carry/mint *decisions* and for the durable graph, but not for `sei_bindings` values. The doc's absolute claim is overstated.

5. **Failure model: plugin findings are log-only, not persisted (MEDIUM drift).** ¬ß6:621‚Äì637 and detailed-design.md ¬ß1077‚Äì1088 specify that every failure emits a structured finding (e.g., `CLA-INFRA-PLUGIN-CRASH`, `CLA-PY-PARSE-ERROR`, `CLA-PY-TIMEOUT`). Code comment at `analyze.rs:626‚Äì628`: "Log findings individually (Tier B persistence is future work)." `log_plugin_findings` (`analyze.rs:2022`) logs them via `tracing::warn!` only. `FINDING_DISABLED_CRASH_LOOP` (`analyze.rs:596`) similarly logs but does not call `InsertFinding`. The "no silent fallbacks" invariant in the doc is not honoured for plugin-level failures.

6. **Crash-loop threshold discrepancy (LOW drift).** ¬ß6:635 documents ">3 crashes in 60s" as the trip condition for `CLA-INFRA-PLUGIN-DISABLED-CRASH-LOOP`. The rule ID in code is `FINDING_DISABLED_CRASH_LOOP` (imported from `clarion_core`), matching the doc intent. However the doc also names a ">10% files ‚Üí CLA-INFRA-ANALYSIS-ABORTED" threshold (`¬ß6:634`, `detailed-design.md ¬ß1088`); the per-file crash-rate breaker is absent in the code.

7. **Resume mechanism differs from checkpoint model (LOW drift).** ¬ß6:605 describes resumption by reading `runs/<run_id>/checkpoints.jsonl` and continuing from the last clean phase boundary. The code (`analyze.rs:173‚Äì174`, `run_lifecycle.rs:25‚Äì34`) implements `--resume` as reopening the run row via `WriterCmd::ResumeRun` and re-running the full pipeline with `mark_unseen=false` on emission. No checkpoint file exists; the skip is provided by Wave 2 incremental file-hash comparison.

The pattern is correct and consistent between Phase 8 and Phase 8b, but it adds a non-obvious indirection; a code comment explains the reasoning. The pattern should be extracted into a shared helper to prevent future call-sites omitting it.

The module doc at `:33` explicitly documents this as v1 scope. A non-Python plugin would get no locator-rename hints from `ShellGitRenameSource`. Low risk at v1; becomes Medium when a second language plugin is added.
