# Rust analysis: known limitations

> New to Loomweave? Start with [getting-started.md](./getting-started.md); this
> page is the reference for what Rust analysis does and does not resolve.

The Rust language plugin extracts entities (functions, structs, enums, traits,
impls, modules) and edges (`contains`, `calls`, `references`, `imports`, plus
the relation kinds `inherits_from`/`implements`/`derives`) from a `.rs` tree.
Several things it deliberately does **not** resolve — by design, by deferral, or
by a tracked defect. This page is the single place to learn which is which, so a
sparse edge set or an unavailable tool reads as "expected" rather than "broken".

Each subsection states **what you see** in the index and **why** it is that way.

## Macro bodies are not expanded

**What you see.** Entities and edges that exist only inside a macro *expansion*
are invisible. A function generated by a `macro_rules!` body, a method a derive
macro synthesises, the call edges inside a `vec![…]` or a custom proc-macro
invocation — none of these appear in the graph. Only the macro *call site* (and
any tokens written literally outside the macro) is indexed.

**Why.** The plugin parses source with `syn` (per spec §5); it reads the code as
written, not as the compiler sees it post-expansion. Expanding macros would
require running proc-macro crates (arbitrary build-time code) and a resolved
build graph, both out of scope for a parse-only, local-first analyzer. The
trade is deterministic, credential-free analysis at the cost of macro-generated
structure.

## External edge targets are dropped by design

**What you see.** Edges whose target lives in *another crate* do not appear. A
`calls` to a `std` or third-party function, an `imports` of an external path, a
`references` to a foreign type — all are dropped rather than recorded as
dangling. Most visibly, **`derives` is empirically sparse**: only derives whose
macro is defined *in this project* are emitted, so a tree that derives mostly
`serde::Serialize`, `Clone`, `Debug`, etc. (all external) will show few or no
`derives` edges.

**Why.** This is Phase-1b decision **D1** (drop-externals): the plugin resolves
edges against the in-project symbol table and discards targets it cannot anchor
to a real in-project entity, rather than minting edges to synthetic
out-of-project nodes. In-project edges stay trustworthy; cross-crate reach is a
later-phase concern. The result is high precision on what is present, with
external relationships simply absent.

## Closures and nested functions are not entities

**What you see.** A closure (`|x| …`) and a function defined inside another
function's body are not their own entities. Calls and references made inside
them are attributed to the nearest enclosing named item, not to a distinct
closure/nested-fn entity.

**Why.** The entity model addresses named, item-level definitions
(functions, structs, enums, traits, impls, modules). Anonymous and locally
nested code has no stable qualified name to key an entity (or an SEI token) on,
so it is folded into its container rather than given its own identity.

## The `references` envelope boundaries

**What you see.** `references` edges cover the common type- and path-mention
sites, but two specific positions are **not** yet emitted: paths bound in
`match` and `let` *patterns*, and the discriminant expressions of enum variants
(`enum E { A = SOME_CONST }`). A reference that appears only in one of those
positions will be missing.

**Why.** These were scoped out of the initial `references` envelope and are
deferred (tracked as **clarion-efc8715d98**). They are an additive expansion of
the same edge kind, not a correctness problem with the edges that are present.

## Parse-guard caps (degrade, never crash)

**What you see.** Before parsing, every `.rs` file is screened by a byte-oriented
guard. If a file trips a cap, it is **not** parsed: it degrades to a single
`module` entity carrying a `parse_status` of `"depth_limit"` or
`"file_too_large"`, plus exactly one **warning** finding naming the measurement
and the cap. The rest of the tree analyzes normally; the plugin never crashes
and the run is not failed.

The caps (ADR-050 §4):

| Guard | Cap | Finding on trip |
|---|---|---|
| Bracket (`(`/`[`/`{`) nesting depth | **128** | `LMWV-RUST-DEPTH-LIMIT` (WARN) |
| Consecutive prefix-operator run (`!`, `&`, `*`, `-`) | **1024** | `LMWV-RUST-DEPTH-LIMIT` (WARN) |
| On-disk file size | **10 MiB** | `LMWV-RUST-FILE-TOO-LARGE` (WARN) |
| Parse-thread stack (internal) | **16 MiB** | — (defence-in-depth, not a cap you trip) |

**Why.** `syn` 2.x has no recursion limit — deeply nested input overflows the
parse stack and SIGABRTs the process. The caps sit ≥4× under the lowest measured
crash floor, so a hostile or machine-generated file costs one degraded entity
plus a visible warning instead of failing the whole run. The guard is a real
lexer (it skips comments, strings, raw strings, and char literals), so banner
comments and bracket-laden string literals do not false-positive: the
**measured false-positive rate is zero** across ripgrep, serde, tokio, and
rust-analyzer. A genuinely 129-deep nest or a 1025-long operator run is the only
thing that trips a cap, and that is reported, not silently dropped.

## Watchdog deadlines

**What you see.** The plugin host enforces three wall-clock deadlines over the
plugin lifecycle. A plugin that hangs past one is killed; the run records a
timeout finding rather than wedging forever.

| Phase | Default | Env override |
|---|---|---|
| Handshake (whole-repo `initialize`) | **300 s** | `LOOMWEAVE_PLUGIN_HANDSHAKE_TIMEOUT_MS` |
| Per-file extraction | **120 s** | `LOOMWEAVE_PLUGIN_FILE_TIMEOUT_MS` |
| Shutdown | **10 s** | `LOOMWEAVE_PLUGIN_SHUTDOWN_TIMEOUT_MS` |

Each override is in milliseconds. A handshake or per-file timeout makes the run
terminal (`failed`); a shutdown timeout leaves a completed run `completed` (the
data was already durable) and only logs a warning finding.

**Why.** ADR-050: the Rust plugin builds its whole-project symbol table inside
`initialize`, so the handshake budget scales with repo size (300 s covers a
~10 M-LOC repo at `syn`'s parse throughput); per-file and shutdown work have
much smaller legitimate envelopes. The deadlines bound damage from a hung or
hostile plugin without policing the throughput of a legitimate one. Raise an
override only if a real repo/analyzer trips it in practice.

## No dead-code signal on a pure-Rust index

**What you see.** `entity_dead_list` returns nothing useful on a project indexed
**only** by the Rust plugin — there is no categorisation surface to draw from.
This is a known limitation, not an error.

**Why.** The dead-code (and related categorisation) views are driven by
tags the language plugin emits; the Rust plugin does not yet emit those
categorisation tags, so a pure-Rust index has no data to populate them
(tracked as **clarion-e1899a109f**). The structural tools (`entity_find`,
`entity_callers_list`, `entity_neighborhood_get`, and the edge surfaces) are
unaffected.

## Unnamed `const _` items are not entities

**What you see.** A `const _: T = …;` item — the deliberately-unnamed
compile-time-assertion idiom — does not appear in the graph at all: no entity,
no `contains` edge, no `references` edges from its type or initializer.

**Why.** `_` is non-identifying by construction: the item is un-nameable, so
nothing (a `use`, a path expression, an issue binding) can ever target it, and
no discriminant could give repeated `const _` twins stable distinct identities
without churning every sibling on edit. ADR-049 Amendment 9 therefore skips
them entirely — deliberately absent, not lost. Named consts are unaffected.

## `#[path]`-mounted modules carry their mount name

**What you see.** A file mounted via `#[path = "…"] mod name;` is indexed under
its **mounted** logical path, not its on-disk path. In the common
platform-backend idiom (tokio's `src/process/`):

| Source | Indexed module |
|---|---|
| `#[cfg(unix)] #[path = "unix/mod.rs"] mod imp;` → `unix/mod.rs` | `tokio.process.imp@cfg(unix)` |
| `#[cfg(windows)] #[path = "windows/mod.rs"] mod imp;` → `windows/mod.rs` | `tokio.process.imp@cfg(windows)` |
| inline facade `pub(crate) mod unix { use super::imp::*; }` | `tokio.process.unix` |

So the *implementation* module answers to the mount name the source actually
declares (`…imp@cfg(unix)` — the `@cfg(...)` suffix appears only when two
same-name mounts coexist), while the *public facade* keeps the public name
(`…unix`). If you search for the public name you find the facade; the
implementation entities live under the mount name. Files under a mounted
directory re-key with it (`process/unix/orphan.rs` →
`tokio.process.imp@cfg(unix).orphan`).

**Why.** ADR-049 Amendment 8: routing mounted files by their on-disk path made
two source modules collide on one identity (the mounted file and its inline
facade), which was silent data loss. The mounted path is what rustc actually
compiles, so it is the honest identity. Three edges of the rule worth knowing:
a `#[path]` inside an unexpanded macro invocation is invisible (the file
routes by its on-disk path); a `#[cfg_attr(pred, path = "…")]`-delivered
mount is invisible by the same rule — only a literal `#[path]` attribute
mounts, so the file routes by its on-disk path; and a mount whose target lies
outside the declaring crate's `src/` tree is ignored. A `use`-path that names a cfg-twin mount resolves as
external (the `@cfg` suffix is not spellable in a `use`), so such import edges
are absent — the same pre-existing behaviour as for any cfg-twin entity.

## Duplicate locators surface as `LMWV-DUPLICATE-LOCATOR` findings

**What you see.** When two source declarations assemble the same entity
identity (a *locator collision* — e.g. the known open cfg-twin and
trait-/self-type-path families), `loomweave analyze` now emits a project-level
**`LMWV-DUPLICATE-LOCATOR`** finding at **ERROR** severity, naming the
colliding id and both source paths. It appears in `project_finding_list` /
`entity_finding_list` like any other analyze finding. The run still completes:
the store keeps one row per id (last write wins), so the colliding entity is a
chimera of both declarations until the underlying collision family is fixed —
the finding is the alarm, not a block.

**Why.** The writer's upsert-on-id is load-bearing for incremental re-analysis,
so a collision used to be absorbed silently. The detection is host-side and
plugin-agnostic (it covers the Python plugin identically), and it is collision-
family-agnostic: it catches the *next unknown* family in the field, not just
the catalogued ones. Legitimate recurrence never trips it — unchanged re-
analysis, genuine moves (the old file is re-analyzed and drops the id), and the
reconciled module dual-claim shape (an inline `mod` facade plus the
path-derived module, first claim wins) all stay silent. A healthy tree
produces zero of these findings.
