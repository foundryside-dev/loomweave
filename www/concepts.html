Loomweave/Concepts

Concepts

Loomweave’s job is to turn a tree of source files into a queryable graph of entities and edges, then serve it to a consult-mode agent. Everything an agent asks is answered against that graph.

Entities & kinds

An entity is a named thing Loomweave extracted from the source: a function, a class, a module, or a clustered subsystem. Every entity carries a 3-segment locator:

locator — {plugin_id}:{kind}:{canonical_qualified_name}
python:function:auth.tokens.refresh
python:class:http.client.Session
python:module:auth.tokens

The contract (ADR-003, ADR-022) is deliberate: the language plugin owns segments 1 and 3 — the plugin id and the canonical qualified name — and the core never invents kinds. That keeps the address reproducible across runs and portable across tools.

Entity kinds
KindWhat it is
functionA function or method, addressed by its qualified name
classA class definition
moduleA source module (file-level)
subsystemA cluster of modules, produced by Loomweave’s clustering pass — derived, not extracted from source like the three above

Three kinds are extracted directly from source (function, class, module); subsystem is a fourth, derived kind that Loomweave synthesises by clustering modules.

Locator vs. identity — the locator is the address, SEI is identity

The crucial distinction. The 3-segment locator above is derived from the name and module path — so a rename or a cross-module move changes it, and every binding keyed on the old string would be orphaned. The Weft suite once conflated identity with address; the qualname is a fine address and a terrible identity, because the operations developers do most — rename, move — change it.

ADR-038 separates the two. The locator is demoted to the address; SEI (Stable Entity Identity) is the identity — a durable id that survives a rename or move, minted and resolved by Loomweave as the suite’s identity authority. A sibling product binds its facts to the SEI and trusts the link still names the same function tomorrow.

Address
Locator {plugin}:{kind}:{qualname}
Human-legible, derived from name + path, reproducible per-run. Changes when the code is renamed or moved.
Identity

The SEI standard is the suite-wide contract; Loomweave ADR-038 is its token form. Loomweave is the authority/implementer — Wardline, Filigree, Legis, and Charter are consumers that conform.

The edge graph

Entities are connected by typed edges:

Edges
EdgeMeaning
containsStructural nesting — a module contains its classes and functions
callsOne function invokes another (execution flow)
referencesA symbol mentions another without calling it (not execution flow)
importsModule-to-module import (imports_in = who imports me, imports_out = what I import)

The three headline relationships are contains, calls, and references; module imports rounds out the graph at module altitude.

calls edges carry a confidence. The default surface is resolved — unambiguous static edges. Ambiguous static candidates and LLM-inferred edges are excluded unless a query explicitly asks for them, so a clean answer stays clean.

Subsystems

Beyond the raw graph, Loomweave clusters modules into subsystems and persists them as first-class entities. subsystem_member_list(id) lists a subsystem’s modules; entity_subsystem_get(id) is the reverse — given any entity, find the subsystem it belongs to (a function resolves through its containing module).

The consult model — ask a tool, don’t re-grep

Loomweave is a consult-mode tool. It does not write your code or run your tests; it answers questions about the codebase so a coding agent doesn’t re-derive the same structural facts on every turn.

Without Loomweave, an agent answering “who calls this, and what does it touch?” greps the tree, opens files, and reconstructs the call graph from scratch — every time, burning context. With Loomweave, the same question is a couple of tool calls against a graph that was already built once by loomweave analyze:

the consult loop
agent ──▶ entity_at(file, line)               ──▶ "which entity is here?"
      ──▶ entity_neighborhood_get(id)         ──▶ callers, callees, imports, refs
      ──▶ entity_execution_path_list(id, n)   ──▶ bounded call paths
      ──▶ summary(id)                         ──▶ one-paragraph explanation (lazy LLM)

Each answer is structured, paginated where needed, and carries enough metadata (confidence, scope_excludes, freshness) for the agent to know how much to trust it.

Why scope_excludes matters

Static analysis has blind spots — calls made through an attribute receiver, for instance, can’t always be resolved statically. Rather than silently dropping them, Loomweave’s traversal results carry a scope_excludes block naming exactly what was not searched.

The honesty contract
An empty callers list with scope_excludes: ["attribute-receiver-calls"] means “none found among the edges I can see,” never “guaranteed nobody calls this.” That distinction is what lets an agent reason safely about a negative result.

The same discipline governs confidence: graph-traversal tools return only resolved edges by default, and an empty section is never read as a guaranteed true negative.

Enrich-only by design

issues_for(id) (the MCP entity_issue_list) reaches into Filigree, a sibling Weft product, to attach issues to an entity. That binding is strictly enrich-only: if Filigree is unavailable, the tool returns an unavailable envelope instead of failing the call, and Loomweave’s own answers never depend on it.

This is the Weft federation axiom in practice — a sibling may add information to Loomweave’s view, but is never required for Loomweave to make sense. See Loomweave’s role in the federation.

Storage

The whole graph is persisted to a project-local SQLite database at .weft/loomweave/loomweave.db, written by a single writer-actor with a reader pool (ADR-011). There is no mandatory cloud component: Loomweave is local-first, and the only network egress is the LLM provider during summary(id) calls.

Loomweave is part of the Weft Federation — the suite’s code-structure surface and its SEI identity authority. Installable and runnable: a single Rust binary plus a Python language plugin.
