Read the codebase,

Loomweave is a code-archaeology tool. It walks a codebase, extracts entities (functions, classes, modules) and edges (contains, calls, references), and persists the structural graph to a local SQLite store. A consult-mode LLM agent that would otherwise re-explore the tree on every question asks a graph-aware tool instead.

It is the federation’s map and its coordinate system: it turns a codebase into a queryable, identity-stable structural graph — and mints the SEI every cross-tool binding keys on.

Why Loomweave is load-bearing for the suite
Loomweave is the Weft suite’s identity authority. It mints, persists, re-binds, and resolves SEI — the Stable Entity Identity every cross-tool binding keys on, so a Filigree issue or a Wardline taint fact still names the same function after a rename or move tomorrow. What Loomweave owns

Structural truth about the codebase — and, load-bearing for the whole suite, its identity authority. Tool names, route shapes, and the entity-ID scheme are Loomweave’s to define.

Entity extraction

Functions, classes, modules — each addressed by the 3-segment locator {plugin_id}:{kind}:{qualname} the language plugin owns.

The edge graph

contains, calls, references, and module imports — typed edges, with calls carrying a resolution confidence.

Subsystem clustering

A clustering pass groups modules into subsystems, persisted as first-class entities you can navigate to and from.

Briefings & guidance

Lazy one-entity-at-a-time LLM summaries, orientation packets, and operator-authored guidance sheets consult agents read.

content_hash

A BLAKE3 hash of each entity body — the drift signal every cross-tool binding checks against to know if it still points at the same code.

SEI — minting, persistence, resolution, lineage

The keystone. Loomweave mints the durable Stable Entity Identity, persists it across re-analysis, re-binds it through renames and moves, and serves resolve / lineage for the suite.

30-second quick start

Install, walk the corpus, serve the graph. analyze needs no LLM credentials — it is the fastest way to verify the install end-to-end.

bash
# install the binary + the Python language plugin from the GitHub Release
TAG=v1.1.0
curl -L -o loomweave.tar.gz \
  "https://github.com/foundryside-dev/loomweave/releases/download/${TAG}/loomweave-x86_64-unknown-linux-gnu.tar.gz"
tar xzf loomweave.tar.gz
install loomweave-x86_64-unknown-linux-gnu/loomweave ~/.local/bin/
pipx install \
  "https://github.com/foundryside-dev/loomweave/releases/download/${TAG}/loomweave-plugin-python-1.1.0.tar.gz"

# wire the agent, walk the corpus, serve the graph over MCP
loomweave install --path .   # initialises .weft/loomweave/, wires the MCP client + skill + hook
loomweave analyze            # extract entities + edges (no LLM credentials needed)
loomweave serve              # expose the ~40-tool consult-mode MCP surface

A traversal answer, honest about its blind spots

Every graph-traversal result carries a scope_excludes block naming exactly what was not searched — so an empty result is “none found among the edges I can see,” never a guaranteed true negative.

json — entity_neighborhood_get(id) (excerpt)
{
  "entity": "python:function:auth.tokens.refresh",
  "callers": [],
  "callees": ["python:function:auth.tokens._sign"],
  "imports_in": ["python:module:auth.session"],
  "confidence": "resolved",
  "scope_excludes": ["attribute-receiver-calls"]
}
Reads as
“No resolved callers among the edges I can see; calls made through an attribute receiver were not searched.” That distinction is what lets an agent reason safely about a negative result. See Concepts → scope_excludes.

Its role in the greater federation

Loomweave is one member of the Weft Federation — an agent-first family of small, local-first developer tools. The federation runs on one axiom, and Loomweave is its identity authority.

The federation axiom
Each member is authoritative for one domain, solo-useful, meaningfully composable pairwise, and enrich-only — never load-bearing — when composed.
The connective tissue
SEI — Stable Entity Identity
One durable id per code entity, owned by Loomweave. Every cross-tool binding keys on it, so a link survives a rename tomorrow. The SEI standard is the suite contract; Loomweave ADR-038 is the token form. LOCKED · 2026-06-05
Identity vs. address
Locator is the address
The 3-segment {plugin_id}:{kind}:{qualname} is a fine address and a terrible identity — rename and move change it. ADR-038 demoted it to the locator and made SEI the identity.
Representative bindings — every one enrich-only
  • Loomweave ↔ Filigree entity ↔ issue, drift-aware via issues_for / entity_issue_list
  • Wardline → Loomweave taint facts enrich the entity graph (Loomweave hosts the taint-fact store)
  • Legis → Filigree SEI-keyed sign-offs on issues
  • Charter → Loomweave SEI consumer for trace links planned · scaffold-state

Remove any sibling and Loomweave’s own answers keep working — it builds its catalog whether any sibling is present or not. Full 5×5 matrix in federation-map.md; the identity contract in sei-standard.md.
